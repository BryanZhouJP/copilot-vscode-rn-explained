### AI活用提案サマリー

今回の VS Code × GitHub Copilot アップデートの中で、**クラウド・オンプレ基盤担当の IT 技術者**が特に注目すべき活用シナリオをまとめます。

| 推奨 | 機能名 | 活用シナリオ | 期待効果 |
| ---- | ------ | ----------- | -------- |
| ★★★ | コンテキスト圧縮（`/compact`） | Terraform 一括リファクタリングなど長時間エージェント作業でコンテキストを圧縮し、会話を中断せずに継続 | コンテキスト切れによるセッション再起動が不要になり、複雑なタスクを途切れなく完了できる |
| ★★★ | 自動承認スラッシュコマンド（`/autoApprove`） | CI/CDスクリプト生成やインフラコード変換で承認ダイアログなしにエージェント作業を全自動化 | 複数ファイルにまたがる変更で手動承認が不要になり、長時間タスクの完全委任を実現 |
| ★★★ | 長距離 Next Edit Suggestions | 関数リネームや型変更など波及効果のある編集でファイル全体の影響箇所を予測・提案 | コード変更後の連鎖修正が自動提案され、見落としによるバグを未然に防止 |
| ★★☆ | `/create-*` コマンドでカスタマイズ生成 | 会話の中でデバッグ手順を確立したら `/create-skill` でスキル化してチームで再利用 | 属人的なワークフローをスキル化して標準化し、チーム全体の生産性を向上 |
| ★★☆ | エージェントプラグイン（Experimental） | チーム共通のインフラ管理ツール群（MCP・スキル・フック）をプラグインにまとめて配布 | エージェント環境をワンクリックで統一し、チームのオンボーディング工数を削減 |
| ★★☆ | チャットセッションのフォーク（`/fork`） | 大規模変更を試みる前にセッションを分岐し、元の作業を保全しながら別アプローチを安全に検証 | リスクのある変更の実験と本線作業を切り離し、安全なアプローチ比較を実現 |
| ★☆☆ | エージェントデバッグパネル（Preview） | カスタムエージェントの動作不具合時にリアルタイムのイベントログでトラブルシューティング | 問題の原因特定時間を短縮し、カスタマイズの品質向上が加速 |
| ★☆☆ | セッションメモリによるプラン永続化 | Plan エージェントで作成したインフラ構成計画を会話ターンをまたいで保持・継続 | 長期インフラ改善プロジェクトで計画のコンテキストを維持し、一貫性を確保 |
| ★☆☆ | AI 共著者帰属（`git.addAICoAuthor`） | AI生成コードを含むコミットに `Co-authored-by: GitHub Copilot` を自動付与して AI の関与を明示 | Git 履歴で AI 支援の範囲を追跡でき、コードレビューや監査での透明性を確保 |

> **凡例**: ★★★ 今すぐ活用推奨 / ★★☆ 近い将来検討推奨 / ★☆☆ 動向ウォッチ推奨

### 1. コンテキスト圧縮（Context Compaction）+ `/compact` コマンド

| 項目 | 内容 |
| ---- | ---- |
| 対象バージョン | VS Code v1.110（2026年3月4日） |
| カテゴリ | 新機能 |
| 対象コンポーネント | Agent Mode / Copilot Chat |
| **業務活用推奨レベル** | **★★★ 今すぐ活用推奨** |

**長時間のエージェント作業でコンテキストウィンドウが埋まっても、会話履歴をサマリに圧縮して同一セッションを継続できるようになった。**

- 自動圧縮（コンテキスト上限到達時）に加え、チャット入力に `/compact` を入力することで任意のタイミングで手動圧縮を実行できる
- 圧縮時に保持したい情報を自然言語で指示できる（例: `/compact インフラの設計決定だけ残して`）
- ローカル・バックグラウンド・Claude エージェントセッションのすべてで手動圧縮が可能
- コンテキストウィンドウ使用率はチャット入力欄のコントロールアイコンでリアルタイムに確認できる

📘 **用語解説**:
- **コンテキストウィンドウ**: LLM が一度に処理できるトークンの上限。会話が長くなるほど消費が増え、古い情報が切り捨てられる
- **圧縮（Compaction）**: 会話履歴を要点のサマリに統合して文脈を保ちながらトークンを節約する処理

🔗 **情報源**:
- [Visual Studio Code 1.110 Release Notes](https://code.visualstudio.com/updates/v1_110)
- [Context compaction](https://code.visualstudio.com/docs/copilot/chat/copilot-chat-context#_context-compaction)

💡 **AI 活用提案**:
- Terraform 大規模リファクタリングや Kubernetes マニフェスト変換など多ターンタスクで `/compact focus on the infrastructure plan` のように重要情報を指定して圧縮し、エージェント作業を中断なく継続する
- コンテキスト使用率が 70〜80% を超えたタイミングで手動圧縮を行い、予期せぬセッション切断を未然に防ぐ運用習慣を取り入れる

### 2. 自動承認スラッシュコマンド（`/autoApprove` / `/yolo`）

| 項目 | 内容 |
| ---- | ---- |
| 対象バージョン | VS Code v1.110（2026年3月4日） |
| カテゴリ | 新機能 |
| 対象コンポーネント | Agent Mode |
| **業務活用推奨レベル** | **★★★ 今すぐ活用推奨** |

**チャット入力から `/autoApprove` または `/yolo` を入力するだけで、設定画面に移動せずにグローバル自動承認を即座に切り替えられるようになった。**

- `/autoApprove` でグローバル自動承認を有効化、`/disableAutoApprove`（または `/disableYolo`）で無効化できる
- 有効化中はファイル編集・ターミナルコマンド実行・外部ツール呼び出しのすべての確認ダイアログがスキップされる
- ターミナルサンドボックス（`chat.tools.terminal.sandbox.enabled`）と組み合わせることで、自動実行しながらコマンドの影響範囲を制限できる
- **注意**: 破壊的操作も無確認で実行されるため、対象スコープを明示的にプロンプトへ指示することが必須

📘 **用語解説**:
- **グローバル自動承認**: すべてのツール呼び出しを自動許可にする設定。個別ツールの承認設定より優先される

🔗 **情報源**:
- [Visual Studio Code 1.110 Release Notes](https://code.visualstudio.com/updates/v1_110)
- [Auto-approve tools](https://code.visualstudio.com/docs/copilot/chat/chat-agent-mode#_auto-approve-tools)

💡 **AI 活用提案**:
- 多ファイルにわたる Terraform/Ansible コードの一括変換タスクで `/yolo` を有効化し、手動承認なしにエージェント作業を連続実行する。完了後は必ず `/disableYolo` で安全な状態に戻す習慣をつける
- ターミナルサンドボックスと組み合わせて「コマンド実行は許可するが本番系ドメインへのアクセスはブロック」のような柔軟なセキュリティポリシーを構築する

### 3. 長距離 Next Edit Suggestions（Long-distance NES）

| 項目 | 内容 |
| ---- | ---- |
| 対象バージョン | VS Code v1.110（2026年3月4日） |
| カテゴリ | 機能強化 |
| 対象コンポーネント | コード補完 |
| **業務活用推奨レベル** | **★★★ 今すぐ活用推奨** |

**カーソル周辺だけでなくファイル全体の任意の行に次の編集箇所を予測・提案する、NES の長距離対応版が実装された。**

- 関数リネーム・パラメータ型変更など「波及効果のある変更」で、ファイル内のすべての影響箇所を自動提案する仕組み
- 専用の位置予測モデルと既存の編集生成モデルを組み合わせたマルチモデルアプローチで、無駄なジャンプを抑制しながら関連箇所を絞り込む
- A/B テストで NES 経由のコード記述量が **23% 増加** を確認（RLVR による強化学習で「ジャンプしない判断」も学習済み）
- カーソル付近にコンパクトなウィジェットで提案が表示され、差分プレビューを確認してから `Tab` でジャンプ・適用できる
- 有効化には `github.copilot.nextEditSuggestions.enabled` と `github.copilot.nextEditSuggestions.extendedRange` の両方を `true` に設定する必要がある

📘 **用語解説**:
- **Next Edit Suggestions (NES)**: ゴーストテキストの拡張版。カーソル位置への挿入だけでなく、次に変更すべき箇所の編集を予測して提案する機能
- **extendedRange**: NES の提案範囲をカーソル周辺からファイル全体に広げる設定フラグ

🔗 **情報源**:
- [Visual Studio Code 1.110 Release Notes](https://code.visualstudio.com/updates/v1_110)
- [Building Long-Distance Next Edit Suggestions](https://code.visualstudio.com/blogs/2026/02/26/long-distance-nes)

💡 **AI 活用提案**:
- インフラコードで変数名・リソース名の命名規則を変更する際に長距離 NES を活用し、ファイル内のすべての参照箇所を Copilot の誘導で一括修正する作業フローを試してみる
- 大規模リファクタリング前に `extendedRange` をオンにして、変更の波及範囲を長距離 NES の提案数で事前把握するインスペクションとして利用する

- [Agent customization](https://code.visualstudio.com/docs/copilot/customization)

- チームのインフラデバッグ手順（ログ確認→原因特定→修正フロー）を会話の中で確立したら `/create-skill` でスキル化し、`.github/skills/` にコミットしてチーム全体に展開する
- プロジェクト固有の命名規則・コーディング規約を Copilot に学習させた後、`/create-instruction` で指示ファイルに保存してリポジトリに追加することで、新メンバーのオンボーディングを自動化する

### 6. チャットセッションのフォーク（`/fork`）

| 項目 | 内容 |
| ---- | ---- |
| 対象バージョン | VS Code v1.110（2026年3月4日） |
| カテゴリ | 新機能 |
| 対象コンポーネント | Copilot Chat |
| **業務活用推奨レベル** | **★★☆ 近い将来検討推奨** |

**会話の任意の時点からセッションを分岐し、会話履歴を引き継いだ新しい独立セッションを作成できるようになった。**

- インフラ構成の大規模変更を実施する前に `/fork` でセッションを分岐し、元のセッション（現行構成の文脈）はそのままに、フォーク先で変更案の検討を進める。採用する案が決まったら元のセッションに知見を持ち帰る運用が有効

### 9. AI 共著者帰属（AI co-author attribution for commits）

| 項目 | 内容 |
| ---- | ---- |
| 対象バージョン | VS Code v1.110（2026年3月4日） |
| カテゴリ | 新機能 |
| 対象コンポーネント | その他（ソース管理） |
| **業務活用推奨レベル** | **★☆☆ 動向ウォッチ推奨** |

**AI 生成コードを含むコミットに `Co-authored-by: GitHub Copilot` トレーラーを自動付与する設定が追加された。**

- `git.addAICoAuthor` 設定値:
  - `off`（デフォルト）: トレーラーを付与しない
  - `chatAndAgent`: Chat・Agent Mode で生成したコードのコミットに付与
  - `all`: インラインコード補完を含むすべての AI 生成コードに付与
- VS Code 内でのコミットにのみ適用。外部 Git ツールやコマンドラインのコミットには影響しない
- Git blame ホバーに co-author 情報が表示されるようになった

🔗 **情報源**:
- [Visual Studio Code 1.110 Release Notes](https://code.visualstudio.com/updates/v1_110)

💡 **AI 活用提案**:
- 組織のコンプライアンスポリシーで AI 支援の範囲の記録が求められる場合、`git.addAICoAuthor: "chatAndAgent"` を設定しておき、Git ログで AI の関与を追跡できる体制を整える

### 習得ガイド 1: コンテキスト圧縮（Context Compaction）

#### 機能詳細解説

- VS Code v1.110 から、長時間のエージェントセッションでコンテキストウィンドウが埋まっても、会話履歴を要点のサマリに圧縮することで同一セッションを継続できるようになった
- 自動圧縮（コンテキスト上限到達時）に加え、`/compact` コマンドで手動でも圧縮を実行できる
- IT インフラ担当者の典型的なユースケース: Terraform/Ansible の大規模一括変換、Kubernetes マニフェストの Cluster 間移行など、多ファイル・多ターンにわたるリファクタリングセッション
- `/compact` の後にカスタム指示を追加することで、圧縮後も残したい情報を指定できる（例: `/compact 現在の Terraform 変更案とリソース命名規則の決定だけ残して`）
- ローカル・バックグラウンド・Claude エージェントセッションのすべてで手動圧縮が可能
- コンテキストウィンドウの使用率はチャット入力欄のコントロールアイコンでリアルタイムに確認できる
- 秘密情報（APIキー・パスワード等）が会話に含まれる場合、圧縮サマリへの漏出を避けるよう指示に注意する

#### 前提条件

- **VS Code バージョン**: v1.110 以上
- **GitHub Copilot プラン**: すべてのプラン
- **必要な設定**: 特になし（デフォルト有効）
- **OS**: Windows（本ガイドは Windows 環境での手順を前提とする）

#### セットアップ手順

特別なセットアップは不要。v1.110 以降であれば自動コンパクションはデフォルトで有効。

1. **コンテキスト使用率の確認**
   チャット入力欄の左側にあるコンテキストウィンドウコントロールアイコン（使用率が % 表示されている箇所）をクリックして使用率を確認する

2. **手動圧縮の実行（基本）**
   チャット入力欄に以下を入力して Enter を押す:
   ```
   /compact
   ```

3. **手動圧縮の実行（保持情報を指定）**
   重要な設計決定を確実に保持したい場合:
   ```
   /compact 現在の Terraform 変更案・リソース命名規則・ネットワーク設計の決定事項を必ず保持して。変換済みコードの詳細は省略可
   ```

4. **圧縮内容の確認**
   圧縮後にチャット上に表示される要約メッセージを確認し、重要な情報が保持されているかをチェックする

> 💡 **ヒント**:
> - コマンドパレット: `Ctrl+Shift+P` → `Chat: Compact Conversation` でも実行可能
> - 使用率が 80% を超えたあたりで手動圧縮するとセッション継続がスムーズ

#### 使い方（ハンズオン）

**シナリオ**: 50 個の Terraform ファイルを AWS から Azure に移行する作業でエージェントと長時間会話しており、コンテキストが限界に近づいている

**この機能がない場合**:
- コンテキストが埋まると古い情報が切り捨てられ、エージェントが以前の設計決定を忘れてしまう
- 「先ほど決めた命名規則は？」と毎回再確認が必要になり、会話の流れが途切れる
- 最悪の場合、新しいセッションを開始して最初からコンテキストを説明し直す必要がある

**操作手順**:

1. Chat ビューのコンテキストウィンドウコントロールで使用率を確認する（70〜80% が目安）
2. チャット入力に以下を入力して手動圧縮を実行する:
   ```
   /compact 以下の決定事項を必ず保持してください：
   - リソース命名規則: {プロジェクト名}-{環境}-{リソース種別}
   - タグ必須項目: environment, owner, cost-center
   - ネットワーク設計: VNet CIDR 10.0.0.0/16、サブネット分割方針
   それ以外の変換済みファイルの詳細は省略可
   ```
3. 圧縮完了後、エージェントに「今の命名規則を確認して」とテスト質問を送り、設計決定が保持されていることを確認する
4. 問題なければ移行作業を継続する

**期待される結果**:
- エージェントが圧縮された要約をもとに設計決定を正しく認識し、後続の会話で一貫した回答を返す
- コンテキストウィンドウの使用率が大幅に低下し、長時間の作業継続が可能になる
- 新しいセッションを開始する必要がなく、会話の流れを中断させずに作業が続けられる

#### ベストプラクティス（IT インフラ業務向け）

- `/compact` を実行するタイミングをルール化する: コンテキスト使用率 80% または「設計決定フェーズが一段落したとき」を目安にするとよい
- 圧縮指示には「必ず保持する情報」と「省略して良い情報」を明示的に分けて記述することで、圧縮後の情報精度が向上する
- 重要な意思決定の直後に手動でサマリを明示的に指示する `/compact` を実行し、決定内容がサマリに確実に含まれるようにする習慣を持つ
- 秘密情報（認証キー・パスワード等）を会話に含めないことが原則。もし含んでしまった場合は圧縮指示で「機密情報は除外して」と明記する

### 習得ガイド 2: 自動承認スラッシュコマンド（`/autoApprove` / `/yolo`）

#### 機能詳細解説

- VS Code v1.110 より、チャット入力から `/autoApprove`（または `/yolo`）を入力するだけで、設定 UI や settings.json を開くことなくグローバル自動承認を即座に有効化できる
- 自動承認有効中はファイル編集・ターミナルコマンド実行・外部ツール呼び出しのすべての確認ダイアログがスキップされる
- ターミナルサンドボックス（`chat.tools.terminal.sandbox.enabled`）と組み合わせると、コマンド実行を許可しながら特定ドメインへのネットワークアクセスをブロックするなど、エージェントの動作範囲を精緻に制御できる
- v1.111 で追加された Autopilot の前段階となる機能であり、エージェントの自律実行を実現する重要な操作起点
- セキュリティリスク: 本番環境へのファイル変更・システム破壊コマンドも無確認で実行されるため、対象スコープを明示的にプロンプトに指示することが必須

#### 前提条件

- **VS Code バージョン**: v1.110 以上
- **GitHub Copilot プラン**: すべてのプラン
- **必要な設定**: 特になし（デフォルトは無効）
- **OS**: Windows（本ガイドは Windows 環境での手順を前提とする）

#### セットアップ手順

1. **自動承認の有効化**
   Chat ビューの入力欄に以下を入力して Enter を押す:
   ```
   /autoApprove
   ```
   または `/yolo` でも同じ効果:
   ```
   /yolo
   ```
   初回実行時はセキュリティ警告ダイアログが表示される。内容を確認の上で承認する。

2. **自動承認の無効化**
   作業完了後は必ず以下で無効化する:
   ```
   /disableAutoApprove
   ```
   または `/disableYolo` でも可。

3. **ターミナルサンドボックスとの組み合わせ（推奨）**
   ```json
   // settings.json に追加
   "chat.tools.terminal.sandbox.enabled": true
   ```
   サンドボックス有効化状態で `/autoApprove` を使うことで、コマンド実行は自動化しながらシステムレベルの操作に制限を設けられる。

> 💡 **ヒント**:
> - `Ctrl+Shift+P` → `Chat: Disable Auto Approve Tools` コマンドでも無効化できる
> - 現在の自動承認状態はチャット入力欄のステータス表示で確認できる

#### 使い方（ハンズオン）

**シナリオ**: 50 個の Kubernetes マニフェスト（YAML ファイル）のネームスペース名を一括置換し、その後 `kubectl apply --dry-run` で構文検証するタスクをエージェントに委任する

**この機能がない場合**:
- エージェントがファイルを変更するたびに「Allow」ボタンを押す必要があり、50 ファイル分の操作が続く
- `kubectl apply` のターミナルコマンド実行も毎回確認が必要で作業が細切れになる
- 連続する確認ダイアログへの対応中、別の作業に集中できない

**操作手順**:

1. Chat ビューで `/autoApprove` を入力して自動承認を有効化する
2. 以下のプロンプトをエージェントに渡す:
   ```
   このワークスペース内のすべての Kubernetes マニフェスト（*.yaml）を確認し、
   namespace: old-namespace をすべて namespace: new-namespace に変更してください。
   変更後、kubectl apply --dry-run=client -f . を実行してエラーがないか確認してください。
   エラーがあれば自動的に修正してください。
   ※ prod および production を含む namespace は変更しないこと。
   ```
3. エージェントが確認ダイアログなしに全ファイルの変更と検証を自動実行するのを確認する
4. 作業完了後、`/disableAutoApprove` で自動承認を無効化する

**期待される結果**:
- 50 ファイルの変更と kubectl dry-run 検証が中断なく自動実行される
- エラーが発生したファイルはエージェントが自律的に修正し、再検証を行う
- 完了時に変更ファイル数・検証結果のサマリが返される

#### ベストプラクティス（IT インフラ業務向け）

- `/autoApprove` を使用する際は必ず「対象外のファイルやディレクトリ」をプロンプトで明示する（例: `prod/ ディレクトリは変更しないこと`）
- 作業前に Git でブランチを切るか `git stash` を実行してチェックポイントを作成し、エージェントの変更を安全にロールバックできる状態にしておく
- 作業完了後は `/disableAutoApprove` を実行することをルール化する（セッションをまたいで有効状態が持続しないよう警戒する）
- セキュリティポリシーが厳格な環境ではターミナルサンドボックスを前提条件とし、サンドボックスなしでの `/autoApprove` 使用を禁止するガイドラインを設ける

### 習得ガイド 3: 長距離 Next Edit Suggestions（Long-distance NES）

#### 機能詳細解説

- VS Code v1.110 で、Next Edit Suggestions（NES）がカーソル周辺の数行以内だけでなく、ファイル全体の任意の行に次の編集箇所を予測・提案できるようになった
- 専用の「位置予測モデル」を採用したマルチモデルアーキテクチャで、「ジャンプすべき場所」の予測と「編集内容の生成」を分離することで精度と速度を両立
- A/B テストで NES 経由のコード記述量が **23% 増加**（RLVR による強化学習で「ジャンプしない判断」も最適化済み）
- 提案はカーソル付近のコンパクトなウィジェットとして表示される。差分プレビューで変更内容を確認してからジャンプ・適用できる
- NES の積極度（eagerness）は Copilot ステータスバーアイコンから調整可能。提案が多すぎる場合は低く設定する
- 有効化には 2 つの設定:
  1. `github.copilot.nextEditSuggestions.enabled: true`（NES 本体の有効化）
  2. `github.copilot.nextEditSuggestions.extendedRange: true`（長距離対応の有効化）

#### 前提条件

- **VS Code バージョン**: v1.110 以上
- **GitHub Copilot プラン**: すべてのプラン（GitHub Copilot が有効であること）
- **必要な設定**: `github.copilot.nextEditSuggestions.enabled` と `github.copilot.nextEditSuggestions.extendedRange` を両方 `true` に設定
- **OS**: Windows（本ガイドは Windows 環境での手順を前提とする）

#### セットアップ手順

1. **設定の有効化**
   ```json
   // settings.json に追加
   "github.copilot.nextEditSuggestions.enabled": true,
   "github.copilot.nextEditSuggestions.extendedRange": true
   ```
   または、`Ctrl+Shift+P` → `Preferences: Open User Settings (JSON)` で settings.json を開いて追記する。

2. **積極度（eagerness）の調整**
   右下の Copilot ステータスバーアイコンをクリック → 「Next Edit Suggestions Eagerness」で調整する。最初は低めに設定して必要な場面だけ提案が来るようにすると作業の邪魔になりにくい。

3. **動作確認**
   任意のソースファイルで関数名を変更し、ファイルの別の場所にある同じ関数名の呼び出し箇所に向けてウィジェットが表示されることを確認する。

> 💡 **ヒント**:
> - 提案ウィジェットが表示されたら `Tab` キーで確認・適用、`Esc` キーで却下できる
> - `Ctrl+Shift+P` → `GitHub Copilot: Configure Next Edit Suggestions` で設定変更も可能

#### 使い方（ハンズオン）

**シナリオ**: Terraform モジュールの変数名 `vpc_id` を `network_id` にリネームし、ファイル内のすべての参照箇所を一括更新したい

**この機能がない場合**:
- `Ctrl+H` での検索置換は変数名と同じ文字列をすべて置換するため、意図しない文字列まで変更するリスクがある
- 手動で全ファイルを確認して置換漏れがないかチェックする必要があり、時間がかかる
- 変数リネーム後に output ブロック・module 呼び出し側の変更が漏れてバグが発生することがある

**操作手順**:

1. settings.json で `nextEditSuggestions.enabled` と `nextEditSuggestions.extendedRange` が `true` になっていることを確認する
2. エディタで変数定義行の `vpc_id` にカーソルを置き、`network_id` に変更する
3. 変更後、カーソル付近にコンパクトウィジェットが現れ、ファイル内の別の参照箇所への提案が表示されることを確認する
4. `Tab` キーを押して提案されたジャンプ先に移動し、提案された変更を `Tab` で適用する
5. 次の参照箇所への提案がある場合は繰り返し `Tab` で適用する

**期待される結果**:
- カーソルが順次ファイル内の参照箇所にジャンプし、各箇所で変更が適用される
- output ブロック・variable 宣言・module 引数など、単純な検索置換では見落としやすいコンテキストの変更も提案される
- ファイル全体のすべての参照箇所が一貫して更新される

#### ベストプラクティス（IT インフラ業務向け）

- IaC コードで命名規則を変更する際のファーストチョイスとして積極的に活用する。単純な `Ctrl+H` 置換より安全で文脈を理解した修正ができる
- `extendedRange` を有効にした状態で大規模リファクタリングを始め、長距離 NES の提案数をもとに変更の波及範囲を把握するインスペクションとして使う
- 提案が多すぎて邪魔に感じる場合は eagerness を下げる。逆に提案が少なすぎる場合は上げるといった調整で自分の作業スタイルに合わせる
